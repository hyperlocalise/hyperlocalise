## 规范文件名

锁文件名称固定为 `.hyperlocalise.lock.json`.

`hyperlocalise` 目前无法读取 `i18n.lock` 或其他别名。

将文件保留在项目根目录中，以便 `run` 以及 sync 命令共享相同的检查点状态。

## 锁定文件的用途

将锁文件视为一个 **咨询检查点**不是你的真实来源。

- 可翻译内容的权威来源是你的源文件。
- 生成输出的真实来源是你的目标文件。
- 同步远程状态的真实来源是你的 TMS 加上本地工件。

锁文件只记录 CLI 已经尝试或完成的内容，以便跳过重复工作。

## 架构

当前形状：

```json
{
  "adapter": "crowdin",
  "project_id": "12345",
  "last_pull_at": "2026-01-15T12:34:56Z",
  "locale_states": {
    "es-ES": {
      "revision": "rev_abc123",
      "updated_at": "2026-01-15T12:00:00Z"
    }
  },
  "run_completed": {
    "apps/web/lang/es-ES.json::checkout.submit": {
      "source_hash": "7f4f7d..."
    }
  }
}
```

字段含义：

- `adapter`: 活动的 TMS 适配器名称。
- `project_id`: 适配器-特定项目标识符。
- `last_pull_at`：最新一次成功同步拉取的时间戳。
- `locale_states`: 每-语言区域检查点映射。
  - `revision`：用于增量拉取逻辑的适配器修订版本或游标。
  - `updated_at`：上次刷新该区域设置检查点的时间。
- `run_completed`: 每-使用的任务完成映射 `run` 用于跳过决策。
  - `source_hash`：在完成时刻捕获的源指纹。

## 按命令划分的生命周期

### `hyperlocalise run`

- 读取 `.hyperlocalise.lock.json` 在规划任务时。
- 跳过已标记为完成的工作于 `run_completed`.
- 将成功的任务持久化回 `run_completed`.

### `hyperlocalise sync pull`

- 读取现有适配器和区域设置检查点元数据。
- 更新 `adapter`, `project_id`, `last_pull_at`，并且 `locale_states` 成功拉取后。

### `hyperlocalise sync push`

- 可能会从锁文件中读取适配器/项目上下文。
- 不取代源/目标内容的权威性。

## 安全重置指南

当检查点状态已过期或不再可信时，重置锁文件。

常见重置场景：

- 你严重更改了源文本结构
- 你更改了存储桶映射或文件路径模板
- 你已切换适配器或项目
- 你怀疑检查点条目已损坏或被手动编辑

安全重置步骤：

1. 确保你的源文件和生成的文件已提交或已备份
2. 删除 `.hyperlocalise.lock.json`
3. 运行 `hyperlocalise run --dry-run` 查看更大的计划
4. 运行 `hyperlocalise run` 和/或 `hyperlocalise sync pull` 用于重建检查点状态

重置后的预期副作用：

- `run` 可能会重新-执行之前跳过的任务
- 下一次 sync pull 可能会对各个区域设置执行更全面的刷新
- CLI 运行时间可能会暂时增加，直到检查点重建完成

## 团队约定

- 不要交给他人-除非在调试本地事件，否则不要编辑 lockfile 字段。
- 在 CI 脚本中明确处理锁文件。
- 当 run/sync 行为看起来异常时，请在拉取请求中检查 lockfile 的更改。
