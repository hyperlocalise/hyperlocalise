<Note>
Hợp đồng API việc làm công khai có thể phát triển trong khi các luồng công việc CI dần ổn định.
</Note>

## Cách sử dụng

```bash
hyperlocalise sync pull [--config <path>] [flags]
```

## Hành vi mặc định an toàn

`sync pull` mặc định ở chế độ chạy thử. Hãy bắt đầu bằng cách xem lại báo cáo, rồi chạy lại với `--dry-run=false`.

`sync pull` đọc một manifest công việc hoàn chỉnh, thăm dò API Hyperlocalise công khai theo ID công việc, tải xuống các tệp đầu ra thành công và ghi chúng vào các đường dẫn đích được phân giải bởi `i18n.yml`.

Manifest thường được truyền từ một công việc GitHub Actions `sync push` sang một công việc `sync pull` với `actions/upload-artifact` và `actions/download-artifact`.

## Cờ

- `--config`: đường dẫn cấu hình
- `--locale`: ngôn ngữ mục tiêu, có thể lặp lại
- `--manifest`: đường dẫn đến tệp manifest của các công việc
- `--wait`: chờ các công việc đang xếp hàng hoặc đang chạy hoàn tất
- `--timeout`: thời gian chờ tối đa, ví dụ `20m`
- `--dry-run`: chỉ xem trước (mặc định `true`)
- `--output`: `text`, `json`, hoặc `markdown`

## Ví dụ

```bash
hyperlocalise sync pull --manifest .hyperlocalise/jobs.json --wait --timeout 20m --dry-run=false
```

```bash
hl pull --manifest .hyperlocalise/jobs.json --dry-run=false
```

## Ghi chú

- `sync pull` từ chối các manifest có `complete: false`.
- Không có `--wait`, các công việc đang xếp hàng hoặc đang chạy sẽ gây ra mã thoát khác 0.
- Các tệp đầu ra được ánh xạ theo ngôn ngữ sang các đường dẫn đích được ghi lại trong tệp kê khai.
- Các lệnh TMS cụ thể theo nhà cung cấp như `crowdin` và `phrase` giữ nguyên cấu hình và hành vi riêng của chúng.
